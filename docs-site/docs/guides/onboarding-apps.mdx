# Onboarding an app

For the consumer apps to be able to integrate with Verified Orchestartion platform, they will need to be onboarded as App registrations in Microsoft Entra ID tenant of Verified Orchestration platform.

An administrator of Verified Orchestration platform can help onboarding an app in Microsoft Azure Portal [[doc](https://learn.microsoft.com/en-us/azure/healthcare-apis/register-application)].

The following outlines the steps needed to create a new App registration in Microsoft Azure Portal.

1. provide the name and leave other options as per default
1. create a client secret with appropriate expiry
1. add relevant API permissions according to the [access model](#access-models) required by the app
1. securely communicate the Application (client) ID and the client secret to the consumer app team

## Access models

Primary access to Verified Orchestration platform is via a client secret. A second access model caters for apps which do not have the ability to securely store a secret. Limited access tokens are used in this case.

### Limited access token model

This model is best suited for the client apps which do not have the ability to securely store a secret such as Single Page Applications (SPA), mobile apps, etc. The following application permissions from Verified Orchestration API are applicable to this access model.

- **VerifiableCredential.AcquireLimitedAccessToken.AnonymousPresentations**<br/>
  Acquire limited access token: anonymous presentations
- **VerifiableCredential.AcquireLimitedAccessToken.Issue**<br/>
  Acquire limited access token: issue
- **VerifiableCredential.AcquireLimitedAccessToken.ListContracts**<br/>
  Acquire limited access token: list contracts
- **VerifiableCredential.AcquireLimitedAccessToken.Present**<br/>
  Acquire limited access token: present

For more information on how the limited access token works, refer to the [Limited access tokens guide](./limited-access-tokens/).

### Conventional access token model

This model is also available for the apps which can securely store a secret such as APIs, automation workflows, etc. The following application permissions from Verified Orchestration API are applicable to this access model.

- **VerifiableCredential.Issue**<br/>
  Credential issuance
- **VerifiableCredential.Present**<br/>
  Credential presentation

Either or both of those permissions can be granted to the App registration depending on the requirements of the consumer app.

## CORS origins

If you are accessing the Verified Orchestration API from a web application, CORS origins will need to be configured against each instance (sandbox and production).

You may want to match subdomains via regex expressions to reduce the configuration required, e.g. `^https://([a-z0-9]+[.])+yourcompany.com$`

For your sandbox, you may want to allow local development origins e.g. `^http://localhost:\d+$`

Send the list of origins for sandbox and production instances to the Verified Orchestration team.
