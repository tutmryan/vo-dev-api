# Audit

Verified Orchestration plaform creates audit entries for the following event types;

- invocations of GraphQL queries and mutations<br/>
  [sample audit entry for GraphQL query](#sample-audit-entry-for-graphql-query), [sample audit entry for GraphQL mutation](#sample-audit-entry-for-graphql-mutation)

- GraphQL subscriptions<br/>
  [sample audit entry for GraphQL subscription](#sample-audit-entry-for-graphql-subscription)
- callbacks for issuances and presentations from Microsoft Entra Verified ID<br/>
  [sample audit entry for callback Microsoft Entra Verified ID](#sample-audit-entry-for-a-callback-from-microsoft-entra-verified-id)
- revoking credentials of an issuee, a contract, or a user in a background job<br/>
  [sample audit entry for revocation of credentials in a background job](#sample-audit-entry-for-revocation-of-issuances-in-a-background-job)

The platform audit log is avaiable in [Azure Event Hubs](https://learn.microsoft.com/en-us/azure/event-hubs/event-hubs-about) for integration with a System Information and Event Management (SIEM) tool of choice, e.g. [splunk](https://www.splunk.com/en_us/blog/platform/splunking-azure-event-hubs.html), [sumo logic](https://help.sumologic.com/docs/send-data/hosted-collectors/cloud-to-cloud-integration-framework/azure-event-hubs-source/), etc.

Generally, an administrator of Verified Orchestration platform would create a dedicated Shared Access Policy (SAS) and provide the access keys and connection string for each integration with the SIEM tool.
